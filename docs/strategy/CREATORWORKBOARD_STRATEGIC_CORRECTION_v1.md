# CreatorWorkboard — Strategic Correction v1

## Status

This document defines the corrected product truth for CreatorWorkboard.

It replaces the older default framing of CreatorWorkboard as primarily a marketing monitor. The system is still control-first and human-operated, but the actual machine being built is now defined more sharply.

## New product definition

CreatorWorkboard is a human-controlled conversion operations system for creator teams.

Its purpose is to make outside-paywall and inside-paywall conversion work operationally manageable, transferable and measurable while keeping humans in control.

CreatorWorkboard is therefore not primarily:

- a new chat product
- a DM automation product
- an AI-first product
- a generic creator CRM
- a wide marketing dashboard
- a full OF/Fansly suite

CreatorWorkboard is primarily:

- a connection-board
- a control layer around conversation-driven conversion work
- a handoff and routing system
- a context-first operator workspace
- a bridge between outside-paywall and inside-paywall operations

## Core principle

The conversation is part of the operation.
The conversation is not the product itself.

The product owns:

- context
- routing
- handoff
- acknowledgement
- assignee state
- source state
- policy and risk
- next action

Existing chat environments continue to own message transport until proven otherwise.

## What the system must solve

CreatorWorkboard must reduce these operational problems:

- context loss between operators
- scattered external chat environments without one operational truth
- poor outside-paywall to inside-paywall routing
- weak handoff between operator and creator
- unclear next actions per lead, thread or case
- hidden risk or policy constraints
- fragmented source information
- unnecessary reconstruction work before action

## One machine, two routes

CreatorWorkboard serves one operational machine with two routes.

### Route 1 — Existing chat operations

Existing revenue-generating chat environments are connected into the board first.

This route exists to:

- protect current revenue
- reduce operator friction
- improve handoff quality
- improve assignment clarity
- reduce chaos in legacy chat workflows

Route 1 has explicit priority over Route 2.

### Route 2 — Creator paywall operations

Later, the same machine supports creator conversion workflows that cross the paywall boundary for systems such as OnlyFans or Fansly.

This route exists to:

- connect outside-paywall and inside-paywall work
- structure creator/operator collaboration
- keep context and routing above the conversation layer
- support later backoffice expansion without making backoffice the current product core

Route 2 may only expand once Route 1 is operational on the board model.

These are not two separate products.
They are two application routes of the same operations machine.

## Product boundary

### In this phase, CreatorWorkboard is

- human-operated
- control-first
- routing-first
- handoff-first
- context-first
- AI-assisted only where that reduces friction without taking over decisions

### In this phase, CreatorWorkboard is not

- automation-first
- inbox-first
- AI-friend software
- a replacement for all existing chat systems
- a full creator suite
- a broad backoffice platform
- a full OF/Fansly operating system

## Role of AI

AI is supportive, not leading.

AI may later help with:

- sorting
- labeling
- summarising
- prioritising
- preparing work for operators
- signaling risk or urgency

AI must not become:

- the main product claim
- an autonomous conversation engine
- a substitute for human-operated service

## Minimal operational unit

The system needs one minimal controllable work unit.

Working term for the current phase:

- connection card

A connection card should ultimately contain:

- source
- outside-paywall / inside-paywall state
- routing state
- assignee
- latest handoff
- acknowledgement state
- next action
- risk and policy state
- external thread reference

This is not a full chat record.
This is a controllable work card around a conversation-driven conversion workflow.

## Screen logic

The operator should not see only a conversation.

The operator should see:

- who this is for
- where this came from
- which creator or route it belongs to
- whether it is outside or inside paywall
- the current routing state
- the latest handoff
- whether the context was acknowledged
- what the next action is
- what the creator must do
- what the operator must do
- which constraints or risks apply

Operational rule:

- first context
- then route
- then action

## NOW

What belongs in NOW:

- connect existing chat operations into the board
- strengthen handoff and acknowledgement
- strengthen operator workspace context
- model routing state
- model outside-paywall and inside-paywall transitions
- keep humans in the loop
- avoid building a new chat as the starting point

## NEXT

What belongs in NEXT:

- improve intake and feeder logic
- refine routing rules
- improve creator/operator collaboration
- add narrow paywall workflows on the same board model
- add cautious AI assistance for preparation and prioritisation

## LATER

What belongs in LATER:

- deeper OF/Fansly workflows
- richer history and activity layers
- broader backoffice support
- more connectors
- possible conversation layer expansion

A proprietary conversation layer is only justified when external chat systems are proven to be the structural bottleneck after the board model itself is working.

## NOT NOW

Not now:

- a new inbox as the core product
- a standalone DM platform
- AI-first positioning
- autonomous conversation flows
- a full OF/Fansly suite
- a broad analytics-first identity
- automation-first market claims

## Business choice

The first business priority is not inventing a new category.

The first business priority is improving the existing conversion revenue machine with a better operational layer.

That means the correct order is:

1. connect existing chat operations
2. make the connection-board operational
3. stabilise routing, handoff and assignment clarity
4. extend the same machine into paywall creator workflows
5. only much later decide whether a proprietary chat layer is justified
