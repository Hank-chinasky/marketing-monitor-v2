# CreatorWorkboard — Execution Roadmap v1

## Status

This roadmap translates the corrected product truth into build order.

It is explicitly aligned to:

- the strategic correction document
- the rewritten Startdocument
- the aligned SPEC-1

This roadmap replaces any older implicit build order that still assumed a marketing-monitor core.

## Operating rule

Build order follows this rule:

1. protect and improve the existing revenue machine first
2. make the connection-board operational before expanding routes
3. stabilise handoff, acknowledgement and routing before deeper integrations
4. add paywall workflows only on the same machine
5. delay any conversation-layer ambitions until the board itself proves its value

## What is already valid and should be preserved

The following already-built or current-direction slices remain useful and should be treated as part of the new machine:

- creator and channel structure
- scoped assignments
- workspace access control
- creator materials
- handoff save flow
- assignment deactivate / reactivate
- channel and workspace discoverability improvements

These are not side projects.
They are early parts of the connection-board foundation.

## Phase 0 — Foundation already in motion

### Goal

Keep the current monolith stable while reinterpreting existing slices under the new product core.

### Includes

- auth and scoped access
- creators and channels
- assignments
- workspaces
- handoff editing
- acknowledgement direction
- creator materials

### Exit condition

- the current internal cockpit is stable
- no scope confusion remains between old marketing-monitor language and new connection-board language

## Phase 1 — Route 1 control layer backbone

### Goal

Make the existing chat operation workable through a clear control layer before building deeper route logic.

### Build in this phase

- workspace hardening
- acknowledgement v1
- channel-level context completeness
- source context fields where missing
- policy and risk visibility improvements
- quick actions into external systems
- clearer operator flow around handoff and next action

### Do not build in this phase

- a new inbox
- autonomous AI actions
- deep paywall product logic
- backoffice expansion

### Exit condition

An operator can reliably open a creator/channel/workspace, see the latest context, confirm it, and act without reconstructing truth from multiple tools.

## Phase 2 — Connection card minimal model

### Goal

Introduce the minimal operational work unit for the board.

### Build in this phase

- `connection_card` as minimal controllable work unit
- creator linkage
- optional channel linkage
- source type and source label
- external system and external thread reference
- paywall state
- routing state
- assignee
- next action
- risk/policy flags
- lightweight note/handoff connection to the card

### Do not build in this phase

- full message storage
- chat transcript ingestion as product core
- rich timeline engine
- advanced analytics dashboards

### Exit condition

A real piece of conversation-driven work can exist as a board-controlled work card, even if the conversation itself still lives elsewhere.

## Phase 3 — Operator queue and routing

### Goal

Make Route 1 operationally usable at queue level.

### Build in this phase

- operator home / queue
- filters for route, assignee, status and urgency
- visible stalled / blocked states
- visible missing handoff / missing acknowledgement states
- next-action-oriented queue rows
- creator and channel entry points from the queue

### Do not build in this phase

- broad metrics dashboards as core focus
- automation-first prioritisation
- creator self-service portal

### Exit condition

Operators can work through assigned cards from one queue with less search time and less context loss.

## Phase 4 — Existing chat connectors by reference

### Goal

Connect existing chat operations into the board without rebuilding transport.

### Build in this phase

- external thread reference handling
- per-system quick action links
- source labeling for legacy chat systems
- operational card ownership around those systems
- lightweight connection events

### Do not build in this phase

- native chat replacement
- full sync ambition unless clearly needed
- deep cross-system ingestion before the queue model proves useful

### Exit condition

Existing revenue-generating chat work can be tracked, routed and handed off through the board while still using external systems for transport.

## Phase 5 — Route 1 hardening

### Goal

Turn Route 1 into a stable working machine.

### Build in this phase

- better queue hygiene
- blocked-state clarity
- clearer policy gating
- stronger handoff discipline
- better acknowledgement behaviour
- lightweight operational metrics such as stalled cards, missing handoff and queue age

### Exit condition

The existing chat operation is measurably calmer, more transferable and easier to run than before.

## Phase 6 — Route 2 narrow paywall expansion

### Goal

Start supporting creator paywall operations on the same board model.

### Build in this phase

- paywall-state-specific routing
- narrow OnlyFans/Fansly context support
- creator/operator collaboration steps
- route-specific next-action patterns
- source-to-paywall transition clarity

### Do not build in this phase

- a full OF/Fansly suite
- broad backoffice
- financial operations platform
- a second independent product architecture

### Exit condition

The same board can support a narrow outside-paywall to inside-paywall creator workflow without becoming a full suite.

## Phase 7 — Assisted enrichment

### Goal

Add only the forms of AI assistance that reduce friction without taking over decisions.

### Build in this phase

- summary assistance
- label assistance
- priority suggestions
- operator preparation support

### Do not build in this phase

- autonomous replies
- hidden automation
- AI-led operator replacement

### Exit condition

AI improves operator preparation without becoming the product’s operating logic.

## Phase 8 — Expansion options

### Only consider after previous phases prove themselves

Potential later expansions:

- more connectors
- richer history layers
- broader paywall operations
- backoffice support
- possible conversation-layer ownership

### Rule for a future conversation layer

A proprietary conversation layer is only justified if:

- Route 1 is already working on the board
- routing and handoff are stable
- external chat systems remain a proven bottleneck
- the business is already paying for the workflow value, not just for a new chat window

## NOW / NEXT / LATER summary

### NOW

- Route 1 first
- workspace + handoff + acknowledgement + context discipline
- connection card minimal model
- queue and routing basics
- external thread references by linkage, not by full transport replacement

### NEXT

- queue hardening
- connector refinement
- operational metrics
- narrow Route 2 expansion

### LATER

- deeper OF/Fansly route support
- broader backoffice
- more connectors
- assisted enrichment expansion
- possible conversation layer

## Not now

- new inbox as core product
- autonomous AI conversation system
- full OF/Fansly suite
- broad analytics-first dashboard identity
- multitenant SaaS expansion
- replacing all transport systems before the board model is proven

## Success condition for this roadmap

This roadmap is successful when CreatorWorkboard first becomes the board that makes existing conversation-driven conversion work calmer and more transferable, and only then expands into deeper creator paywall operations.
